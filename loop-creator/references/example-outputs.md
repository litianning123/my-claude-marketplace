## Tier 1: Command

**Scenario:** "Monitor CI status for PR #42 every 15 minutes"

```
/loop 15m Monitor CI status for PR #42.

Context: Latest CI run logs from GitHub Actions
Action: Classify the result as passing, failing, or in-progress
Stop when: Notify me and stop after one check
Verify: CI status was correctly read and matches the actual build result
Do not modify any code or files.

(This is a polling interval. The loop checks every 15m and reports only when something changed.)

```

**Manual test reminder (always included):**

```
## Before scheduling
Run this manually once first:
1. Paste the command without `/loop <cadence>` into Claude Code
2. Verify the output is correct
3. Check that no unexpected files were modified
4. Only then schedule with the full `/loop` command above

```

## Tier 2: Project

**Scenario:** "Daily project review every morning"

### TASK.md

```markdown
# Daily Project Review Loop

## Goal
Daily project review

## Expected Output
Each run should produce or update:
outputs/daily-review.md

## Scope
Inspect the project workspace, check for new or modified files, identify blockers. Do not modify source files.

```

### LOOP_INSTRUCTIONS.md

```markdown
    # Loop Instructions

    You are running a 24h loop.

    ## Before You Start
    1. Read `TASK.md`.
    2. Read `PROGRESS.md`.
    3. Inspect the relevant context for this run.
    4. Identify what changed, what is incomplete, and what needs human review.

    ## What You Should Do
    Inspect the workspace, summarize what changed since the last run, identify blockers or unresolved questions, write a structured daily review to outputs/daily-review.md, and update PROGRESS.md.

    After completing the action, update `PROGRESS.md` with:
    - Date of this run
    - Summary of what happened
    - Files or areas checked
    - Output produced
    - What the next run should do
    - Anything that needs human review

    ## Safety Rules
    - Do not delete files
- Do not rename or move files
- Do not modify source files
- Only write to outputs/daily-review.md and PROGRESS.md
    - If you are unsure whether an action is allowed, stop and ask for human review.

    ## Scheduled Run Policy
When this loop runs on a schedule:
- If there are meaningful changes, write a concise report.
- If there are no meaningful changes, write a short "No meaningful changes" note.
- If human review is needed, mark it clearly in `PROGRESS.md`.
- If the same blocker appears in two consecutive runs, do not keep retrying. Escalate to human review.
- If verification fails twice, stop and mark the run as not accepted.
- Keep scheduled outputs short unless there is something important to report.
- Do not create extra files unless explicitly allowed.


    ## Verification Checklist
    Before ending the run, verify the following:
    - outputs/daily-review.md exists
- Summary section present
- Files reviewed section present
- Blockers section present
- Recommended next actions section present
- PROGRESS.md was updated
- No files outside allowed paths were modified

    ### Safety Boundary
    Only the files explicitly listed above may be modified.
    If any other file was modified, stop and report the issue.

    ## Failure Policy
    If verification fails:
    1. If the failure is a missing section or output, fix it once.
    2. If `PROGRESS.md` was not updated, update it once.
    3. If any forbidden file was modified, stop immediately and report the issue.
    4. If the same verification check fails twice, stop and mark the run as needing human review.

    ## Stop Policy
    Stop after writing the report and updating PROGRESS.md. Escalate to human review if the same blocker appears in two consecutive runs.

```

## Tier 3: Skill

**Scenario:** "Daily project review, reusable across my team's repos"

```markdown
---
name: daily-project-review
description: "Use when the user wants a recurring daily review of their project to track changes, identify blockers, and produce a structured summary. Trigger phrases: daily review, project review, morning check, daily standup prep."
---

# Daily Project Review

## Overview
A repeatable loop for: Daily project review.

## When to Use
- When the user asks for: daily review, project review, morning check, daily standup prep
- When the task repeats on a schedule
- When the output can be verified against a checklist

## File Structure
Create this workspace before the first run:

```
<loop-name>/
├── TASK.md
├── LOOP_INSTRUCTIONS.md
├── PROGRESS.md
└── outputs/
    └── <output-file>.md
```

Templates for each file are in the reference implementation.

## Operating Procedure
1. Read TASK.md and PROGRESS.md
2. Inspect the workspace for changes since the last run
3. Write a structured daily review to outputs/daily-review.md
4. Update PROGRESS.md with this run's details
5. Run the verification checklist
6. If no meaningful changes, write a short note and stop

## Verification Checklist
Before ending the run, verify:
- Output file exists
- All required sections are present
- PROGRESS.md was updated
- No files outside allowed paths were modified

## Safety Rules
- Do not delete, rename, or move files
- Do not modify source files
- Only write to the explicitly allowed output paths
- Mark anything uncertain for human review

## Manual Before Scheduled
Run the loop manually 3-5 times before scheduling. Check:
- Output is useful and concise
- State file is updated correctly
- Safety boundaries are respected
- Verification passes consistently

## Scheduling
Once manual runs are stable:
```
/loop <cadence> Run the Daily project review loop. Follow LOOP_INSTRUCTIONS.md exactly.
```

## Common Mistakes
- Scheduling before manual testing
- Skipping state file update at end of run
- Letting output grow too verbose
- Forgetting to check verification before marking done
- Treating first successful run as proof of reliability

## Reference
See `reference-implementation/` for a complete working example.

```
